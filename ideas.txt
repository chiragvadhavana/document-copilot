embedding: 
- images: image bytes + bounding box with bge-base. (caption, vague/generic caption, no caption, text nearby etc)
- image quality pre-filter: aspect ratio, one color, etc. using cheap gemini flash model. classify each image, drop junk categories. write "display caption" for that image. 
- display caption: generating using gemini flash model. it's display only, not used for search.
- dual embed: image stored as two vectors, not one. caption + page vector. 
future: embed the display caption + raw caption + page vector. vector + image both in one page.
- vector images: (due to not getting images from get_images) if lines/curves/paths are more than 40 and text is less than render page as a image using pixmap. text is always chunks regardless. sepp 21 images.
- pdf -> page text chunk -> images -> page render fallback -> image quality filter -> embed with bge base. 
- ingest: embed all chunks -> build faiss indexflatip index. save 3 files in s3. index.faiss, vectors.f32, metadata.json. 
- query: download vectors.f32 + metadata.json from S3 -> load into memory -> brute force JS loop -> return top-k.
- opensearch ingest: embed all chunks -> put vectors to opensearch index (OS stored them as build hnsw index automatically)
- opensearch query: embed query -> send search req to OS API. -> runs hnsw index -> returns top-k. 
- filtering metadata: there we can filter based on various params like native operations in DB. like type, company etc to reduce the number of entries.

understanding pdfs, and building generic solutions. can use model to understand pdf structure as well.
